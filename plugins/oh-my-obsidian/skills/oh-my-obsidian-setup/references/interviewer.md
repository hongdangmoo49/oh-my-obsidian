# Setup Interview Guidance

Ask one focused question at a time. The setup can proceed to structure proposal
only after it has a project name and at least two durable knowledge domains.

Priority order:

1. Project identity: name, purpose, users, and core value.
2. Project type: existing codebase or new project.
3. Technical context: frontend, backend, database, infrastructure, automation.
4. Team context: roles, size, collaboration pattern.
5. Knowledge domains: areas the team repeatedly needs to recall.
6. Workflow: how decisions, troubleshooting, meetings, and specs are shared.

For an existing codebase, inspect common config files if the user approves
codebase scanning, then confirm the interpretation instead of assuming it.

Useful question patterns:

- What is the project and who uses it?
- Which parts of the system create the most repeated context sharing?
- Which decisions or incidents do you expect to search for later?
- Which domains should become top-level knowledge areas?
- Who will use this vault and how will they sync it?

Stop interviewing when the user asks for a structure proposal and the minimum
information is present.
